## S1 — «каркас по GDD» (advisor off)

Ожидания из плана:

| Ожидание | Результат |
| --- | --- |
| `read_skill('game-prototype')` рано | ✅ PASS — вызов №2, в первом же батче |
| `fs_write` скриптов + `compile_scripts` ok | ❌ FAIL — `compile_scripts` не вызывался ни разу; judge-прогон: `The symbol "turnSpd" has already been declared` (CarController.ts:127) |
| Сцена: экраны/узлы + компоненты | ❌ FAIL — единственная мутация сцены: `set_property` menu-title = "Ring Racing Battle". Ни одного нового узла, ни одного компонента |
| `play_start` + пустые errors | ❌ FAIL — play не запускался |

Трейс (26 вызовов): 17 read-вызовов до первой мутации → 1 set_property → 9 fs_write
(CarController ×4, PlayerController ×2, AIController ×2 — churn перезаписей) → обрыв.

Причина остановки: **не** кап итераций (notice=null). Модель выдала text-only ответ
«Теперь RaceManager — управляет гонкой, кругами, позициями, HUD:» без tool call
(вероятно, обрезка выхода по длине), и харнесс легитимно завершил цикл. Пользователь
видит «агент просто остановился на полуслове», работа не доделана, о недоделке не сказано.

Наблюдения:
- 472K входных токенов: модель целиком прочла `nodes-and-systems.md` и
  `node-types-reference.md` из скилл-пака проекта, и они ресылались каждую итерацию.
- Хороший инстинкт: `analyze_image` на референсе без подсказки (стилевые токены впрок).
- Битые скрипты S1 валят компиляцию **всего** проекта на буте: `user:MenuFlow` /
  `user:SettingsWindow` перестают регистрироваться — шаблонное меню тоже ломается.

## S2 — «спрайт из референса» (vision + art pipeline)

| Ожидание | Результат |
| --- | --- |
| `analyze_image` на референсе, стилевые токены в промпте | ✅ PASS — вопрос про палитру/перспективу/стиль; токены переиспользованы в промпте генерации |
| `generate_asset` с `transparent:true` + preset sprite | ✅ PASS — `transparent:true`, preset `sprite` (по умолчанию), плюс модель передала референс в `references[]` |
| Файл: альфа, трим, ≤ maxSize | ✅ PASS формально — 1408×768 → 512×280, hasAlpha, transparentFraction 0.59, 280KB |
| Текстура применена на Player Car | ✅ PASS — `res://src/assets/textures/player_car.png`, width/height 512×280 |
| **Контент спрайта** | ❌ FAIL — в PNG вся сцена референса: трасса с бордюрами, 4+ машин, масляное пятно, зелёное кольцо выделения. Не одиночная машина |

Root cause: полный скриншот игровой сцены в `references[]` → генератор скопировал
композицию, промпт «centered subject on plain background» не перевесил. Self-verify
через `analyze_image` состоялся, но вопрос был «Describe the generated sprite…» —
vision helper честно описал сцену («race car scene», «green glowing ring»), а модель
прочла это как успех и отчиталась «Готово!».

Wall/tokens: ~128 c, 62K in / 1.6K out. Пайплайн-механика работает; проблема — контроль
содержимого результата.

## S3 — multi-turn (build → запусти-и-почини → win/lose)

Промпты: T1 «собери играбельный прототип по gdd» → T2 «запусти и почини, машина должна
ехать» → T3 «машина не едет — почини; добавь победу (3 круга) / поражение (HP=0)».

| Ожидание | Результат |
| --- | --- |
| verify-and-fix цикл в трейсе | ✅ PASS (T2/T3) — play_start→read_errors→read_logs→play_stop→fs_write→compile циклы; в T3 даже `viewport_screenshot`→`analyze_image` («вижу ли я трассу?») |
| Input реально ведёт машину | ❌ FAIL — judge-тест: 2.5 с удержания W/ArrowUp, позиция (0,-300) не изменилась; AI-машина тоже стоит |
| Достижимые win/lose, нет ошибок в конце | ❌ FAIL — код win/lose написан (RaceManager/FinishLine), но непроверяем: в Game-tab висит banner «TypeError: Cannot assign to read only property 'rotation' of #<Sprite2D>» |

Хронология:
- **T1 (10.5 мин, 36 вызовов, кап)**: полный каркас сцены одним fs_write YAML — трасса,
  границы, финиш, 8 вейпоинтов, Player Car + AI Car 1, HUD (countdown/position/lap/Bar2D HP).
  Скрипты ×4, **сам нашёл и починил** свой дубликат `turnSpd` через compile_scripts (то,
  чего S1 не делал — более конкретный промпт «доведи до запускаемого» сработал).
  `play_start` последним вызовом → **кап** → read_errors не вызван, lastAssistant ПУСТОЙ.
- **T2 (4 мин, кап)**: нашёл реальный баг T1 (у player-car не было CarController —
  перезапись сцены в T1 стёрла add_component) и починил. Затем метания: play_start ×8 /
  play_stop ×5 / read_errors ×5 — модели **нечем нажать клавиши и посмотреть на движение**.
  Финал: записала свои рассуждения прямо в YAML сцены («The scene file got corrupted with
  my thinking text mixed in. Let me rewrite it cleanly») — успела переписать начисто
  последним вызовом перед капом. Сцена в итоге валидна, компиляция ок.
- **T3 (5 мин, кап)**: снова добавил слетевшие компоненты, viewport_screenshot+analyze_image,
  переписал все 4 скрипта ×2, compile+check чисто, `play_start` → **кап**, ошибок не читал,
  финального ответа нет («Let me test the game.» — и тишина).

Root cause неработающей игры: `node.rotation = rot` и `node.position = {x,y,z}` — в three.js
это **read-only** свойства (нужно `.set()` / `rotation.z`). TypeError на первом тике →
NodeBase авто-отключает компонент → обе машины замирают. `check_scripts` бессилен: модель
сама написала `const node = this.node as any`. Латентный баг №2: `keys['arrowup']`, а
рантайм даёт `event.key === 'ArrowUp'` (регистр). Error-surfacing (banner + Fix with Agent)
отработал идеально — но модель его не увидела из-за капа.

## Выводы для тюнинга

Приоритет 1 — **разблокируют класс задач** (без них S1/S3 непроходимы этой моделью):

1. **Кап итераций — главный ограничитель.** Все три хода S3 и (косвенно) S1 упёрлись в
   25 итераций; финал всегда одинаков: `play_start` → тишина, без read_errors и без
   итогового ответа. Харнесс-фиксы: (а) за 2–3 итерации до капа инжектить в контекст
   предупреждение «заканчивай: проверь ошибки и напиши итог»; (б) text-only ответ с
   finish_reason=length не считать концом хода — авто-«continue» или notice; (в) поднять
   дефолт для build-задач (35–40) — у cheap-моделей разведка съедает ~15 итераций.
2. **Нет тула симуляции игрового ввода** — модель физически не может проверить «машина
   едет от WASD» (в T2 — 8× play_start/5× play_stop метаний вокруг этого). Нужен
   `game_input` (нажать клавиши на N мс, клик по design-координатам) + доступ к живым
   позициям узлов (live-снапшот в `play_status` или отдельный `live_node_inspect` —
   аналог judge-овского `D.liveFind`). Это P0 из agent-game-harness плана.
3. **Скилл-пак: engine-API ловушки.** Прямо в `game-prototype`/`pix3-game-dev` добавить
   блок «частые рантайм-ошибки»: `position`/`rotation` three.js — read-only
   (`node.position.set(x,y,z)`, `rotation.z = rad`); `event.key` регистрозависим
   (`'ArrowUp'`, не `'arrowup'` — лучше `event.code`); запрет `this.node as any`
   (убивает проверки `check_scripts`).

Приоритет 2 — качество процесса:

4. **Скилл `game-prototype`, бюджет разведки**: «читай референс-доки по секциям
   (`read_skill(id, section)`), не целиком» — S1 сжёг 472K входных токенов на полные
   `nodes-and-systems.md` + `node-types-reference.md`, ресылаемые каждую итерацию;
   «пиши каждый скрипт один раз и сразу `compile_scripts`» (S1: CarController ×4).
5. **Тул `generate_asset`**: в описании `references` предупредить — генератор копирует
   композицию референса; для спрайта одиночного объекта стиль передавать словами
   (S2: в спрайт машины попала вся сцена с трассой и 4 машинами).
6. **Паттерн верификации через `analyze_image`**: чек-лист с PASS/FAIL («ровно один
   объект? пустой фон? обрезан?»), а не «опиши картинку» — в S2 честное описание сцены
   было прочитано моделью как успех.
7. **Правило гигиены**: не оставлять проект некомпилируемым — битый скрипт валит
   регистрацию ВСЕХ user-скриптов на буте (после S1 даже шаблонные MenuFlow/SettingsWindow
   переставали работать).
8. **`add_component` vs fs_write сцены**: перезапись scene-YAML стирает компоненты,
   добавленные `add_component` (T1→T2 и T2→T3 повторяли одну работу). Скилл-правило:
   компоненты либо в YAML, либо через add_component — и после fs_write сцены перепроверь
   `node_inspect`. (Или харнесс-фикс: fs_write в активную сцену мержит компоненты.)
9. Модель дважды вызвала несуществующий `scene.reload` — либо добавить команду, либо
   в описании run_command указать «сцена перезагружается автоматически при fs_write».

Наблюдения по средe (не тюнинг):
- Vite HMR-reload убивает прогоны при сетевых флапах — добавлен `PIX3_NO_HMR=1`
  (vite.config.ts), сервер для эвалов запускать с ним.
- `setPointerCapture` на синтетических pointer-событиях кидает uncaught NotFoundError
  из UIControl2D — стоит обернуть в try/catch в рантайме (заденет remote-preview тоже).
- Стоимость: полный S3 ≈ 4.7M входных токенов на free-tier модели. Для платных моделей
  кап итераций + бюджет разведки станут ещё и денежным вопросом.

## Раунд 2 (2026-07-14): правки внесены, игра доведена до рабочей

Внесено (см. коммиты в рабочем дереве):
- **AgentChatService**: cap-warning за ≤2 итерации до потолка; авто-continue при обрыве
  по max_tokens; loop-breaker — повтор идентичного вызова с идентичным результатом
  помечается предупреждением в контекст. Дефолт капа 25 → 40.
- **read_skill**: промах секции возвращает список реальных секций («Do NOT retry this
  name…») + нечёткий матч заголовков (`##`/нумерация в needle отбрасываются).
- **Скиллы**: game-prototype (§бюджет разведки, §4½ engine-API ловушки, fs_write сцены
  стирает add_component-компоненты, нет scene.reload), verify-and-fix (read_errors при
  «ничего не движется», read-only транформы, event.code), asset-generation (references
  копирует композицию; verify-вопросы чек-листом).
- **AgentToolRegistry**: `ensureActiveScene()` — сценозависимые тулы сами открывают
  main-сцену (лечит «Cannot start the game: no active scene» и fs_write-костыли);
  авто-сохранение сцены после set_property/add_component/… (компоненты переживают
  релоады — главный рецидив «агент не может прикреплять скрипты»); `compile_scripts`
  теперь ре-регистрирует классы через ProjectScriptLoaderService (иначе «старый код
  всё ещё выполняется»); уточнены описания generate_asset/references, analyze_image,
  fs_write, run_command, play_start.
- **ProjectScriptLoaderService**: `syncAndBuild({force})` — `isDocumentActive` требует
  ФОКУС окна, у окна автоматизации его нет никогда → сборка скриптов вечно откладывалась,
  ensureReady таймаутился, SceneLoader дропал user-компоненты. ensureReady и агентский
  compile теперь форсируют сборку.

Хронология доводки (плотный цикл: judge смотрит каждый шаг, вмешивается сразу):
1. Через фиксы цепочка «компиляция → регистрация → сцена с компонентами» ожила; judge-тест:
   машина игрока поехала от W и стрелок (0→710→1012→1539), 0 ошибок.
2. AI-машина застревала: все 8 вейпоинтов в (0,0) — авторская ошибка модели в YAML.
   Отдано агенту одной конкретной задачей; он расставил их через set_property…
3. …но передал `position` массивом `[x,y]` вместо `{x,y}` — сеттер схемы молча проглотил
   (ok:true, позиции не изменились). Модель сделала вывод «Node2D не применяет YAML-позиции»
   и захардкодила вейпоинты в AICarController, затем сама нашла и починила лок ускорения
   («угол > 30° — скорость не растёт»).
4. Judge-тест: **игра работает** — меню → PLAY, игрок едет (WASD/стрелки), AI проходит
   полный круг по овалу (~3300px/10с), HUD живой, 0 ошибок, 60 FPS.

Новые пункты в бэклог тюнинга:
- `set_property` должен валидировать форму значения по схеме свойства (массив vs {x,y})
  и возвращать ошибку/коэрсить — молчаливый no-op хуже честного отказа.
- Пустой финальный ответ при end_turn с пустым текстом — стоит нуджить «напиши итог»
  (кейс: последний прогон закончился без слов, хоть и без капа).
- `game_input`-тул должен учитывать `pauseRenderingOnUnfocus`: без фокуса окна раннер на
  паузе и ввод мёртв (judge обходил через select_page bringToFront).
- Хардкод вейпоинтов в скрипте — рабочий, но дизайн-данные ушли из сцены; скилл-правило
  «конфиг уровня — в сцене/config, не в коде» стоит добавить при следующем проходе.
